# Self-Improvement Reflexes

Use this reference when meaningful work reveals a reusable behavior signal that JARVIS OS should notice without waiting for a full R&D loop.

Reflexes make JARVIS feel alive inside a local markdown system: the agent notices small patterns, captures candidates, compresses them, and routes only strong signals toward promotion.

## Reflex Contract

A Reflex is a tiny candidate-producing behavior, not a standing rule change.

It may:

- notice repeated friction, user correction, stale state, missing verification, bloated notes, useful shortcuts, or recurring project conventions;
- create a compact candidate in the project-local reflex inbox, Friction Inbox, learning log, or the smallest canonical memory home;
- mark the candidate with lifecycle lane: `observed`, `candidate`, `repeated`, `promoted`, `rejected`, `archived`, or `superseded`;
- propose promotion into an idea loop, work order, eval, template, or workflow patch.

It must not:

- silently rewrite core rules;
- promote one-off observations into standing behavior;
- mutate public templates without normal verification and closeout;
- capture private details into public-safe artifacts;
- create ceremony for trivial self-contained work.

## Reflex Triggers

Run a quick Reflex check after meaningful work, especially when one of these appears:

- the user corrects behavior, routing, memory, tone, or workflow;
- the same mistake or hesitation appears twice;
- a closeout, blocker, handoff, version, or task-board drift is detected;
- a small shortcut made work notably easier;
- a rule helped repeatedly and may deserve promotion;
- a note, work order, or project memory is becoming hard to retrieve;
- the agent used a new pattern that future agents should reuse;
- a previous rule felt noisy, redundant, or over-applied.

Skip the check for tiny factual answers, routine command output, and local edits with no reusable learning.

## Reflex Lanes

- `observed`: noticed once; no action unless it recurs or is high-impact.
- `candidate`: worth preserving as a tiny note.
- `repeated`: appeared more than once or in more than one project.
- `promoted`: accepted into a work order, template, eval, rule, or project memory.
- `rejected`: reviewed and intentionally not used.
- `archived`: kept only for history; not active.
- `superseded`: replaced by a newer candidate, rule, or artifact.

## Capture Shape

Keep captures small. A good Reflex candidate answers:

- what happened;
- where it happened;
- why it may matter again;
- proposed response;
- privacy lane;
- next review trigger.

If the signal is mostly workflow friction, use the Friction Inbox. If it is a stable user preference, route to personal preferences. If it is project-local, route to project memory. If it affects JARVIS OS itself, route to JARVIS R&D.

## Promotion Rule

Promotion requires one of:

- repeated signal;
- high-impact failure prevention;
- explicit user approval;
- strong evidence from a real project;
- successful dogfood in the R&D idea loop.

Promotion outputs can be:

- idea card;
- work order;
- eval prompt/check;
- template update;
- public workflow patch;
- memory split or cleanup task;
- rejected/archived note with reason.

## Compression And Pruning

Alive systems need digestion.

When Reflex candidates accumulate:

- merge duplicates;
- compress verbose candidates into one operational sentence;
- archive stale candidates that have not recurred;
- reject candidates that add ceremony without reducing future work;
- promote only the few that materially improve future behavior.

Do not let Reflexes become a second task board. They are intake metabolism, not project management.

## Dogfood Evidence

When a Reflex produces an accepted improvement, record:

- trigger observed;
- candidate created;
- promotion path;
- what changed;
- whether the change reduced future work, prevented drift, or improved memory.

This evidence should feed the next R&D loop when it is stronger than pure ideation.
