# R&D Loop

The R&D Loop is a Specialist Mode pattern for improving JARVIS OS itself.

It turns creative ideas into reviewed, implemented, dogfooded workflow changes.

## Loop Shape

1. Run Feature Adoption Check for named or naturally triggered JarvisOS features.
2. Briefly check whether the last R&D feature affected this cycle.
3. Generate one out-of-box idea.
4. Discuss it with the smallest useful role set.
5. Accept, reject, merge, or defer it.
6. Convert accepted ideas into a short task list.
7. Implement.
8. Refresh the skill and public template if needed.
9. Define where the new behavior should be dogfooded next.
10. Return to ideation.

## Feature Adoption Check

Before working, the team names which existing features should matter in this cycle. If Melih names Jarvis Doctor, Usage Example Closeout, or another feature, it must be used or explicitly skipped with a reason. Public template changes should run Jarvis Doctor before closeout.

## Dogfood Return Check

Before a fresh idea, the team can ask:

- What was the last feature supposed to improve?
- Did this cycle use it, avoid it, or show it is too early to judge?
- Did it reduce friction or add overhead?
- Should today's idea use that evidence or ignore it?

Keep this tiny. It is a return signal, not a full retrospective.

## Guardrails

R&D should not add complexity endlessly.

Use:

- Subtraction Gate: what does this simplify, remove, or make less necessary?
- Feature Adoption Check: are named or naturally triggered features actually being used?
- Dogfood Return Check: did the last improvement actually change this cycle?
- Skill Dictionary Update Check: did this cycle create, remove, rename, or materially reroute a skill?
- Project Skill Incubator: should this stay local to one project before becoming global?
- Sunset Clause: when should this be reviewed or retired?
- Sunset Review Ledger: where are active review triggers tracked when they start to scatter?
- Learning Pulse: what recent evidence supports this?
- Regression Rehearsal: does it preserve behavior we already care about?
- Stack Compression Check: can new ideas merge with existing rules?

## Evidence

Good R&D ideas come from:

- real project friction;
- repeated agent failure;
- stale-state incidents;
- public artifact quality problems;
- user feedback;
- successful field patterns.

Synthetic ideas can seed creativity, but real projects should calibrate the workflow.
